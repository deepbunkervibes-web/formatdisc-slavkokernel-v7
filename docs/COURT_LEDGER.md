# COURT: The Monotonic Ledger Specification

The **COURT Ledger** is the immutable repository of all execution proofs generated by the SlavkoKernel.

## I. Ledger Structure

The ledger is a monotonic sequence of **Decision Envelopes** ($D_n$).
Each $D_n$ contains:

```json
{
  "seq": "uint64",
  "nonce": "hex",
  "prev_hash": "sha256",
  "timestamp": "iso8601",
  "decision": {
    "pipeline_id": "string",
    "input_hash": "sha256",
    "output_hash": "sha256",
    "slavko_score": "float [0.0-1.0]"
  },
  "signature": "ecc-ed25519"
}
```

## II. The Monotonicity Constraint

A Ledger $L = \{D_0, D_1, ..., D_n\}$ is valid if and only if:

1. $D_n.prev\_hash = H(D_{n-1})$
2. $D_n.seq = D_{n-1}.seq + 1$
3. $D_n.timestamp > D_{n-1}.timestamp$

## III. Forensic Access (The "Sovereign Node")

Sovereign nodes have read-only access to the Ledger for compliance verification.

- **Verification**: Recalculating the hash chain confirms the integrity of all historical decisions.
- **Audit Outbound**: Exporting the Ledger as a signed ZIP satisfies Section 4 Requirements for high-risk systems.
